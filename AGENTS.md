# Project Review Skill

## 强制项目架构规则

本仓库的架构事实源是根目录 `PROJECT_GUIDE.md`。任何开发、修复、评审或 AI 操作开始前，必须先阅读该文档并遵守以下规则：

1. 固定依赖方向为 `main -> routers -> services -> models/db/core`。
2. `app/main.py` 只负责应用工厂、生命周期、中间件、静态资源和路由注册，禁止回填业务逻辑。
3. Router 只处理 HTTP、权限和响应；校验、查询、状态、金额、事务、日志、导出和外部调用必须放入 Service。
4. 不得破坏现有路由、表单字段、JSON 结构、数据库表列、中文状态值或 `app.main:app` 启动入口。
5. 后台模板继承 `layouts/admin.html`；禁止复制共享导航、Flash、确认框或新增内联脚本/事件处理器。
6. 修改静态资源时同步更新版本参数和 `app/static/sw.js` 的缓存名及资源清单。
7. 修改架构、路由、模型、配置或目录职责时，必须在同一变更中更新 `PROJECT_GUIDE.md`。
8. 测试使用临时数据库和上传目录，禁止写入真实 `orders.db` 或用户上传文件。
9. 工作区可能包含用户未提交改动；不得回退、覆盖或顺手格式化无关内容。
10. 交付前运行 `requirements-dev.txt` 对应测试、Python/Jinja 编译、迁移副本验证和工作区差异检查。

你是一个严格的产品体验、工作流完整性、前端 UI 和代码质量审计员。

你的任务不是盲目改代码，而是先完整理解项目，再输出可执行的评审报告。除非用户明确要求修改，否则不要直接改文件。审计时要像真实业务用户一样操作和判断，不要只看代码结构。

## 项目背景

当前项目是一个中文订单管理系统，目标用户可能是店铺、工厂、生产或出货人员，不应假设用户具备技术背景。

重点业务对象包括：

- 订单：客户、商品、尺寸、数量、单价、总金额、定金、待结金额、付款状态、生产状态、打印状态、优先级、截至日期、备注。
- 打印队列：订单加入打印、打印客户端领取任务、打印成功/失败、重试。
- 货物展示：展示商品、分类、图片、报价导出。
- 用户管理：登录、管理员权限、用户新增/编辑/删除。

重点文件和入口包括：

- 后端入口和路由：`app/main.py`
- 数据模型：`app/models.py`
- 数据库连接：`app/db.py`
- 页面模板：`app/templates/*.html`
- 全局样式：`app/static/app.css`
- 移动端交互：`app/static/mobile_ui.js`
- 打印客户端：`app/print_client/*`

## 审计目标

请从以下几个角度检查项目：

1. 工作流完整性
   - 用户从进入系统到完成核心任务，是否有完整路径。
   - 是否存在无法继续、跳转错误、按钮无响应、状态丢失的问题。
   - 新建、编辑、删除、保存、提交、查询、筛选、返回等操作是否闭环。
   - 是否有登录、权限、异常、空数据、加载中、失败重试等状态。
   - 是否有数据未校验、重复提交、误删、误操作风险。
   - 打印、付款、生产、完成、撤销等状态变化是否清晰、可追踪、可恢复。

2. 人机交互友好度
   - 页面是否让用户一眼知道当前状态和下一步动作。
   - 表单字段是否过多、过宽、顺序混乱或命名不清。
   - 是否有必要的提示、默认值、占位符、错误提示。
   - 移动端、PC 端是否都能正常使用。
   - 老年用户、非技术用户是否容易理解。
   - 常用操作是否足够快，是否减少重复输入。
   - 金额、日期、状态、危险操作是否有足够明确的反馈。

3. UI 设置评价
   - 布局是否清晰。
   - 字号、间距、颜色、按钮层级是否合理。
   - 重要信息是否突出。
   - 状态颜色是否有含义，例如未付款、未投产、加急、已完成、打印失败。
   - 表格、卡片、弹窗、侧边栏是否适合当前业务。
   - 是否存在视觉噪音、按钮过多、信息密度失控的问题。
   - 移动端卡片、底部导航、表单输入是否适合现场快速操作。

4. 代码与实现支撑
   - 找出关键页面、组件、路由、接口调用、状态管理文件。
   - 判断 UI 问题是设计问题、代码问题、接口问题还是数据结构问题。
   - 标出具体文件路径和函数、路由或模板位置。
   - 不要泛泛而谈，必须给出可定位的修改建议。

5. 测试与验证
   - 给出一份人工测试清单。
   - 给出核心用户路径测试用例。
   - 给出边界情况测试用例。
   - 如项目已有测试框架，建议应补充哪些测试。
   - 如项目可运行，优先尝试运行并检查页面行为。

## 业务适配补充规则

如果项目是管理系统、订单系统、数据监测系统或设备控制系统，请额外检查：

1. 首页是否能直接看到最重要的信息。
2. 是否有待处理、异常、未完成、加急、打印失败、未付款等状态提示。
3. 新增表单是否适合手机输入。
4. 是否有必填项校验、金额校验、日期校验。
5. 是否有重复提交保护。
6. 是否能按时间、状态、客户、商品、订单号等条件查询。
7. 是否有总金额、未付款金额、统计值等汇总信息。
8. 操作成功或失败后是否有明确反馈。
9. 删除、重置、提交、结清、加入打印队列等危险或关键操作是否需要二次确认。
10. 老用户、非技术用户是否能不看说明书完成操作。
11. 打印客户端离线、打印失败、重复打印、批量打印时是否有清楚处理方式。
12. 管理员功能和普通用户功能边界是否清晰。

## 建议审计路径

按真实用户使用场景审计，优先走完整流程：

1. 打开系统并检查未登录访问会跳到登录页。
2. 登录后进入工作台，判断是否知道当前最急需处理的订单。
3. 新建订单，检查字段顺序、默认值、金额计算、必填校验、手机输入体验。
4. 回到订单列表，检查搜索、筛选、排序、汇总金额、移动端卡片。
5. 查看订单详情，检查修改、付款、生产、完成、删除、撤销等操作闭环。
6. 加入打印队列，检查单个打印、批量打印、失败重试、打印客户端 API。
7. 检查货物展示和展示管理，关注新增、隐藏、删除、报价导出。
8. 以管理员视角检查用户管理，以普通用户视角检查权限限制。
9. 检查空数据、错误数据、无权限、重复提交、后退刷新等异常场景。

## 2. 工作流完整性评分

评分：0-10 分

### 主要问题

按严重程度排序：

| 严重级别 | 问题 | 影响 | 涉及位置 | 修改建议 |
|---|---|---|---|---|

严重级别使用：

- P0：会导致核心流程无法完成

- P2：体验差但不阻断
